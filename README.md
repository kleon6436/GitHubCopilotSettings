# GitHub Copilot Settings Template

> **このリポジトリはテンプレートです。**  
> 各プロジェクトに応じて、以下のファイルを書き換えてください。
>
> - `copilot-instructions.md` — プロジェクト名・概要・技術スタック・リポジトリ構成
> - `instructions/` 配下の各 `.instructions.md` — プラットフォーム別のコーディング規約
> - `agents/` 配下の各 `.agent.md` — 使用するモデルやカスタム指示（必要に応じて）
> - `skills/` 配下の各 `SKILL.md` — プロジェクト固有のガイドライン（不要なものは削除）

---

## 概要

本テンプレートは、GitHub Copilot のエージェント・インストラクション・スキルを体系化し、AI 駆動の開発ワークフローを標準化するためのものです。

---

## ディレクトリ構成

```
.
├── copilot-instructions.md   # プロジェクト共通の Copilot 指示
├── agents/                   # 専門エージェント定義
├── instructions/             # プラットフォーム別コーディング指示
└── skills/                   # 専門知識スキル集
```

---

## エージェント一覧

12 のエージェントは **オーケストレーション層** と **専門エージェント層** の 2 層で構成されています。

---

### オーケストレーション層

#### `sisyphus` — メインオーケストレーター

| 項目 | 内容 |
|------|------|
| モデル | Claude Sonnet 4.6 |
| 役割 | 全タスクの起点。ユーザーの意図を分析し、適切な専門エージェントへ委譲・統合・検証 |

**主な責務**
- ユーザーの「文字通りの指示」ではなく「本当に望んでいること」を分析
- 必要な場合のみ `BOULDER.md` によるセッション継続管理（マルチセッション作業・複雑な引き継ぎ・明示的な要求時）
- サブエージェント出力の独立検証（盲目的に信頼しない）

**使用例**
```
@sisyphus ユーザーログイン機能を追加したい
@sisyphus このバグの原因を調査して修正してほしい
@sisyphus リファクタリング計画を立てて実装まで進めてほしい
```

---

#### `sisyphus-junior` — 軽量オーケストレーター

| 項目 | 内容 |
|------|------|
| モデル | GPT-5 mini |
| 役割 | タイポ・単純な 1〜3 行変更・設定値変更など、推論不要な小タスクを低コストで処理 |

**対象タスク**
- タイポ / スペルミス修正
- 1〜3 行の単純変更
- 変数名・ファイル名のリネーム
- コメントの追加・削除
- import 文の整理
- 設定値の変更（ロジック変更なし）

**使用例**
```
@sisyphus-junior "recieve" を "receive" に修正して
@sisyphus-junior この定数の値を 100 から 200 に変更して
```

### 専門エージェント層

#### `prometheus` — 戦略プランナー

| 項目 | 内容 |
|------|------|
| モデル | GPT-5.4 mini |
| 役割 | 要件定義・受け入れ基準の策定・実行計画の作成。コードは一切書かない |

**主な責務**
- ユーザーへのインタビューで「何を・なぜ・誰のために・どうなったら完了か」を明確化
- コードベース探索（`explore` エージェントと連携）
- 実装計画ドキュメントの作成（タスク・担当エージェント・依存関係・完了条件を含む）
- 必要に応じて `metis` / `metis-deep` によるギャップ分析や `momus` によるレビューへ回し、計画の確度を高める

**使用例**
```
@prometheus 通知センター機能の要件を整理して実装計画を作って
@prometheus このフィーチャーのスコープを明確にしたい
```

---

#### `hephaestus` — 自律型深層ワーカー

| 項目 | 内容 |
|------|------|
| モデル | GPT-5.3-Codex |
| 役割 | 目標だけを与えれば、探索→計画→判断→実行→検証を自己完結で実施。明示的な起動が必要 |

**5 フェーズワークフロー**
1. **EXPLORE** — コードベースのマッピング（`explore` エージェントを並列起動）
2. **PLAN** — 実装計画の策定
3. **DECIDE** — アプローチ比較・ユーザー確認（変更規模が大きい場合）
4. **EXECUTE** — 計画に忠実な実装
5. **VERIFY** — 完了条件の独立検証

**適用シナリオ**
- 原因不明の複雑なデバッグ
- クロスドメインな統合
- 大規模リファクタリング
- エンドツーエンドの新機能実装

**使用例**
```
@hephaestus 認証システムを OAuth2 に移行してほしい（全フェーズ自律で）
@hephaestus パフォーマンス低下の根本原因を特定して修正して
```

#### `oracle` — トップレベルコンサルタント

| 項目 | 内容 |
|------|------|
| モデル | GPT-5.4 |
| 役割 | 「どうすればいいかわからない」ときに起動。複雑なデバッグとアーキテクチャ決定に特化。コードは書かない |

**起動条件**（これ以外では起動しない）
- 複雑なバグの根本原因が特定できない
- 複数のアーキテクチャ選択肢があり、正解が不明
- 技術的負債の返済戦略が必要

**使用例**
```
@oracle API レイテンシが断続的に急増する原因がわからない
@oracle マイクロサービス vs モノリス、今のタイミングで判断したい
```

---

#### `librarian` — エビデンスベースリサーチャー

| 項目 | 内容 |
|------|------|
| モデル | GPT-5 mini |
| 役割 | 公式ドキュメントや実際の GitHub 事例を調査。URL・パーマリンクなしの回答は禁止 |

**主な責務**
- 公式ドキュメント・公式 GitHub リポジトリ・公式ブログを優先調査
- バージョン情報の確認
- 複数ソースによるクロスリファレンス

**使用例**
```
@librarian Swift Concurrency の structured concurrency の使い方を調べて
@librarian React 18 の Suspense の最新ベストプラクティスを調べて
```

---

#### `explore` — 高速コードベーススキャナー

| 項目 | 内容 |
|------|------|
| モデル | GPT-5 mini |
| 役割 | コードベースを高速にスキャン。並列実行可。読み取り専用（ファイルの編集・作成・削除は不可） |

**主な責務**
- ファイル・関数・クラス・変数の所在を探す
- 依存関係・参照関係のマッピング
- 変更影響範囲の把握

**使用例**
```
@explore AuthManager を使っているファイルをすべて探して
@explore UserRepository の実装パターンを確認して
```

---

#### `metis` — プランコンサルタント（標準）

| 項目 | 内容 |
|------|------|
| モデル | GPT-5.4 mini |
| 役割 | 実装前の計画に潜む曖昧さ・誤った前提・抜け漏れを検出。計画が完全になるまで承認しない |

**ギャップ分析の観点**
- 前提条件の妥当性検証
- エラーケース・境界条件の網羅性
- タスクの依存関係と順序
- 受け入れ基準の検証可能性

**使用例**
```
@metis この実装計画のレビューをして
@metis この設計に抜け漏れがないか確認して
```

---

#### `metis-deep` — プランコンサルタント（複雑系）

| 項目 | 内容 |
|------|------|
| モデル | GPT-5.4 |
| 役割 | 複数サービス・データモデル変更・セキュリティ制約・マイグレーション戦略が絡む複雑な計画の深層分析 |

**起動条件**（標準の `metis` では対応不足な場合）
- 複数サービスにまたがる変更
- データモデル変更（スキーママイグレーション・ロールバック戦略が必要）
- セキュリティ制約・脅威モデルへの影響
- ロールアウト / マイグレーション / フィーチャーフラグ / ロールバック戦略が必要
- オープンクエスチョンが 3 件以上

**使用例**
```
@metis-deep マルチテナント対応へのマイグレーション計画をレビューして
@metis-deep 認証基盤の刷新計画に穴がないか深くチェックして
```

---

#### `momus` — コードレビュアー（標準）

| 項目 | 内容 |
|------|------|
| モデル | GPT-5.4 mini |
| 役割 | コードレビュー・テスト品質評価・セキュリティ確認（OWASP Top 10）。非自明な変更のデフォルトレビュアー |

**3 つの検証軸**
1. **コードレビュー** — 変更が目的と一致しているか、ロジック欠陥はないか
2. **テスト品質評価** — テストが受け入れ基準を本当に検証しているか
3. **セキュリティレビュー** — OWASP Top 10 ベースの標準チェック

**使用例**
```
@momus この PR の変更をレビューして
@momus この実装に問題がないか確認して
```

---

#### `momus-deep` — セキュリティ特化レビュアー

| 項目 | 内容 |
|------|------|
| モデル | GPT-5.4 |
| 役割 | 認証・認可・データ削除・外部入力処理・並行処理・シークレット管理など、セキュリティ境界を超える変更の深層レビュー |

**起動条件**（これ以外では標準の `momus` を使う）
- 認証・認可ロジック
- データ削除・不可逆な操作
- 外部入力処理（ユーザー向け API・ファイルアップロード・フォーム）
- シークレット・API キー・権限管理
- 並行処理・ロック・競合状態のリスク

**使用例**
```
@momus-deep パスワードリセットフローの実装をレビューして
@momus-deep この API エンドポイントのセキュリティを徹底的に確認して
```

---

#### `atlas` — 実装エージェント

| 項目 | 内容 |
|------|------|
| モデル | GPT-5.4 mini（大規模リファクタや既存規約への密な追従が必要な場合は Claude Sonnet 4.6 に切り替え） |
| 役割 | 検証済み計画の実装。CI/CD・デプロイメント対応も担当 |

**主な責務**
- `prometheus` / `sisyphus` / ユーザーが作成した計画に基づく実装
- 既存の命名規則・スタイル・パターンへの準拠
- CI/CD パイプラインの構築・修正
- 実装結果の独立検証

**使用例**
```
@atlas この計画に基づいてログイン画面を実装して
@atlas GitHub Actions の CI パイプラインを追加して
@atlas このバグを修正して（計画と受け入れ基準は添付）
```

## タスク別エージェント早見表

| タスクカテゴリ | 具体的なタスク例 | 推奨エージェント | モデルコスト |
|--------------|---------------|---------------|------------|
| タイポ・単純変更 | タイポ修正、1〜3行変更、設定値変更、import 整理 | `sisyphus-junior` | 低 |
| 要件整理・計画策定 | 要件定義、受け入れ基準の策定、実装計画の作成 | `prometheus` | 中 |
| 計画レビュー（標準） | 単一サービスの実装計画のギャップ分析 | `metis` | 中 |
| 計画レビュー（複雑系） | 複数サービス・DBマイグレーション・セキュリティ要件・ロールアウト戦略を含む計画のレビュー | `metis-deep` | 高 |
| アーキテクチャ判断 | 設計方針の選択、技術的負債の返済戦略、複雑なバグの根本原因調査 | `oracle` | 高 |
| 自律型大規模実装 | 原因不明のデバッグ、クロスドメイン統合、大規模リファクタリング、E2E 新機能実装 | `hephaestus` | 高 |
| 実装・CI/CD | 計画に基づく機能実装、バグ修正、CI/CD パイプライン構築・修正 | `atlas` | 中（※） |
| UI/UX・アクセシビリティ | UI 実装、デザインシステム適用、アクセシビリティ対応 | `atlas`（Gemini 3.1 Pro） | 中 |
| コードレビュー（標準） | コード品質・テスト品質・OWASP Top 10 セキュリティチェック | `momus` | 中 |
| コードレビュー（セキュリティ特化） | 認証・認可・データ削除・外部入力・並行処理・シークレット管理に関わる変更のレビュー | `momus-deep` | 高 |
| コードベース探索 | ファイル・関数・クラスの所在確認、依存関係の把握、変更影響範囲の調査 | `explore` | 低 |
| ドキュメント・ライブラリ調査 | API 仕様の確認、ライブラリのベストプラクティス調査、エラー原因調査 | `librarian` | 低 |
| 複合タスク全般 | 上記に当てはまらない・複数カテゴリにまたがるタスク | `sisyphus` | 高 |

> **※** `atlas` は通常 GPT-5.4 mini を使用。大規模リファクタリングや既存規約への密な追従が必要な場合は Claude Sonnet 4.6 に切り替えを検討。

## エージェント委譲フロー

```
ユーザー
  │
  ├─ 小タスク（タイポ・1〜3行変更）──────────── sisyphus-junior
  │
  └─ それ以外 ───────────────────────────────── sisyphus
       │
       ├─ 要件整理・計画策定 ─────────────────── prometheus
       │     └─ 計画ギャップ分析（標準）──────── metis
       │     └─ 計画ギャップ分析（複雑系）────── metis-deep
       │
       ├─ アーキテクチャ判断・複雑デバッグ ───── oracle
       │
       ├─ 自律型大規模実装 ─────────────────── hephaestus
       │
       ├─ 実装・CI/CD ────────────────────────── atlas
       │
       ├─ コードレビュー（標準）────────────────── momus
       ├─ コードレビュー（セキュリティ特化）──────── momus-deep
       │
       ├─ コードベース探索 ──────────────────── explore
       └─ ドキュメント・ライブラリ調査 ─────────── librarian
```

## カスタマイズガイド

各プロジェクトへの適用時に書き換えが必要な箇所です。

### `copilot-instructions.md`

| 項目 | 内容 |
|------|------|
| `{Project Name}` | プロジェクト名 |
| `{Brief description...}` | プロジェクトの概要 |
| `{iOS / Android / Web / Server, etc.}` | ターゲットプラットフォーム |
| `{Monorepo / Single repo...}` | リポジトリ構成の説明 |
| 技術スタックテーブル | 使用言語・フレームワーク・ツール |

### `instructions/` 配下

プラットフォームに応じて使用するファイルを残し、不要なものは削除してください。

| ファイル | 対象 |
|----------|------|
| `ios.instructions.md` | iOS アプリ開発 |
| `android.instructions.md` | Android アプリ開発 |
| `macos.instructions.md` | macOS アプリ開発 |
| `web.instructions.md` | Web フロントエンド開発 |
| `windows.instructions.md` | Windows アプリ開発 |
| `cross-platform.instructions.md` | クロスプラットフォーム開発 |

### `skills/` 配下

プロジェクトで必要なスキルのみ残し、不要なものは削除してください。

## BOULDER.md について

`sisyphus` エージェントは、マルチセッション作業・複雑な引き継ぎ・明示的な要求など、**セッション継続の価値がある場合に限って**、プロジェクトルートの `BOULDER.md` を生成・更新します。  
一回で完結する作業では `BOULDER.md` を使わず、その存在自体にトークンを割かない方針です。Git 管理の対象にするか否かは、プロジェクトの方針に応じて決定してください。
